Key Management

Generate, backup, and restore P-256 ECDSA key pairs. Your private key controls your address and funds.

Concepts

Private Key
A 32-byte P-256 (secp256r1) ECDSA private key. This is the secret that signs transactions and proves ownership. Never share it.
Public Key
Derived from the private key. Used to verify signatures. Shared freely — it's included in transactions so validators can check signatures.
Address
The CID (content identifier) of the public key. This is your on-chain identity — balances are stored at account:<address> in the state trie.

Why P-256? P-256 (also called secp256r1 or prime256v1) has hardware acceleration on Apple Silicon, Intel, and ARM chips via Secure Enclave / hardware security modules. This means signing and verification are fast and can be hardware-backed.

Generate a key pair

Using the CLI

lattice-node keys generate --output ~/.lattice/miner.key

This generates a new P-256 private key and writes it to the specified file. The output also shows your public key and address:

Private key written to: ~/.lattice/miner.key
Public key:  04a1b2c3d4e5f6...  (hex, uncompressed)
Address:     baguqeera...abc

Show key info

lattice-node keys info --key ~/.lattice/miner.key

Displays the public key and address for an existing private key file without exposing the private key itself.

Multiple keys

You can generate as many key pairs as you want. Common patterns:

  • Miner key — receives block rewards. Keep on the mining machine.
  • Hot wallet key — for day-to-day transactions. Small balance.
  • Cold storage key — for long-term holdings. Generated offline, backed up securely.

Backup

If you lose your private key, your funds are gone forever. There is no recovery mechanism, no support team, no reset. Back up your keys before mining or receiving funds.

File backup

The key file is a single file containing your P-256 private key. Copy it to secure storage:

# Copy to encrypted USB drive
cp ~/.lattice/miner.key /Volumes/SECURE/lattice-keys/miner.key

# Verify the copy
shasum -a 256 ~/.lattice/miner.key /Volumes/SECURE/lattice-keys/miner.key

Hex export

Export the raw private key as hex for paper backup or hardware wallet import:

lattice-node keys export --key ~/.lattice/miner.key --format hex
Private key (hex): 3a7f2b...
Write this down and store securely. Do not photograph or screenshot.

Backup checklist: Store backups in at least two physically separate locations. Test restoration from your backup before relying on it. Encrypted storage is strongly recommended.

What to back up

  • Private key file — the .key file is all you need. Everything else (public key, address) is derived from it.
  • Not needed — chain data, node config, peer lists. These can be re-synced.

Restore

From file

Copy the backed-up key file to your node's data directory:

cp /Volumes/SECURE/lattice-keys/miner.key ~/.lattice/miner.key

# Verify it works
lattice-node keys info --key ~/.lattice/miner.key

From hex

Import a hex-encoded private key back into a key file:

lattice-node keys import --hex "3a7f2b..." --output ~/.lattice/restored.key

Verify restoration

After restoring, confirm the address matches your original:

lattice-node keys info --key ~/.lattice/restored.key
# Address: baguqeera...abc  ← should match your original

# Check on-chain balance
curl http://localhost:8545/api/balance/baguqeera...abc

Security best practices

  • File permissions — restrict key files to owner-only: chmod 600 ~/.lattice/*.key
  • Separate keys — don't use the same key for mining and cold storage
  • Offline generation — for high-value keys, generate on an air-gapped machine
  • Hardware backing — on macOS, P-256 keys can be backed by Secure Enclave for signing without key extraction
  • Rotation — periodically move funds to fresh keys, especially if a machine was compromised

How signing works

When you send a transaction, the node:

  1. Constructs a TransactionBody with your account actions, fee, and nonce
  2. Stores the body in CAS, producing a bodyCID
  3. Signs bodyCID with your P-256 private key (ECDSA)
  4. Wraps signature + body reference into a Transaction
  5. Broadcasts to the network

Validators verify the signature against the public key, then check that the public key's CID matches the signer address in the transaction. This proves you control the address without revealing the private key.
